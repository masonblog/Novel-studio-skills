# 流水线各阶段详细流程与质量门禁

## 阶段 0：init（总编）

1. 解析用户创意：题材、卖点、目标篇幅、风格偏好、更新诉求。信息不足时**只问一轮**，集中提问（篇幅档位、批次大小、是否要美术物料、风格基调），其余合理默认。
2. 创建工作区目录与 `project.yaml`（按模板），`stage: planning`。

篇幅档位默认值：

| 档位 | 总章数 | 章均字数 | batch_size |
|------|--------|----------|------------|
| 短篇 | ≤10 | 2000–3000 | 不分批（一批完成） |
| 中篇 | 11–40 | 2500–3500 | 5 |
| 长篇 | 41–200 | 3000–4000 | 5 |
| 超长篇 | >200 | 3000–4000 | 10，且按卷分期立项 |

## 阶段 1：planning（总编）

产出 `01-planning/plan.md`：作品定位、各阶段任务分派、批次划分、汇报节点、风险与应对。
**门禁**：将计划要点汇报用户并获确认（自动化/无人值守场景下，用户已在最初指令中给足约束时可视为预确认，但须在汇报中说明）。

## 阶段 2：architecture（架构师）

顺序产出：`world.md` → `synopsis.md` → `outline.md`。
**门禁**（总编核验）：

- [ ] 世界观无内在矛盾，力量/规则体系有明确边界与代价
- [ ] 梗概含完整起承转合与结局，主线闭环
- [ ] 大纲覆盖全部章节，每章有目标、冲突、钩子
- [ ] 伏笔表完整：每条伏笔有埋设章与回收章

## 阶段 3：lore（设定师）

基于大纲产出 `03-lore/` 设定卡。
**门禁**（总编核验）：

- [ ] 主角、反派、所有戏份≥3章的配角均有人物卡
- [ ] 主线任务与各支线有任务卡，与大纲章节对应
- [ ] 大纲中出现的关键场景、关键道具、关键情节均有卡
- [ ] 卡内信息与 world/synopsis/outline 无冲突

## 阶段 4–5：drafting / editing 分批循环

每批流程（批大小 = `project.yaml.batch_size`）：

1. **总编**更新 `progress.current_batch`，列出本批章节与对应大纲、设定卡清单。
2. **写手**逐章起草至 `04-drafts/batch-NN/`，每章自检（字数、钩子、视角）后交接。
3. **编辑**逐章润色对齐，定稿入 `05-final/`，写修订记录；发现设定冲突报总编。
4. **总编**写 `reports/batch-NN-report.md` 并向用户汇报：本批章节概要、字数统计、修订要点、下一批预告、待用户决策事项。
5. **用户确认**后 `status: approved`，进入下一批。用户要求调整时：总编登记 decision → 指派架构师/设定师改上游 → 必要时写手返工 → 重新汇报。

**门禁**：批内每章通过编辑的一致性核对清单（见 `novel-editor` 技能）。

## 阶段 6：art（设计师）

默认在全部正文定稿后执行；用户要求时也可在每批定稿后增量产出该批插画（与下一批写作并行）。
**门禁**（总编核验）：

- [ ] 封面规格文档 1 份（含提示词、构图、字体排版建议）
- [ ] 每个主要角色 1 份立绘规格
- [ ] 每个关键情节卡对应 1 份插画规格
- [ ] 形象描述与人物卡/场景卡一致（发色、服饰、标志物等）

## 阶段 7：acceptance（总编）

1. 按交付清单逐项核对：全章定稿、字数总计、设定文档、美术物料、修订记录。
2. 产出 `reports/final-report.md`：作品简介、数据统计、目录、文件索引、遗留事项。
3. 向用户交付，`stage: delivered`。

## 中断恢复

任何角色启动时：读 `project.yaml` → 确定 stage 与 current_batch → 检查该阶段产物完整性（半成品文件以实际内容为准补齐）→ 从断点继续。不重做已过门禁的阶段。
